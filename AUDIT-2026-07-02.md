## 🔴 Sécurité

### S1 — Injection de formule tableur dans les exports CSV — gravité HAUTE — ✅ corrigé (5ddd1ea)
- **Constat** : `tri._write_segment_csv/_write_invalids_csv/_write_phone_csv` et
  `drafts.export_mailmerge` écrivent les valeurs de la base acquise telles quelles.
  Une cellule commençant par `=`, `+`, `-`, `@`, TAB ou CR (ex. `nom="=cmd|' /C calc'!A0"`)
  est exécutée comme formule à l'ouverture dans Excel/LibreOffice — et l'humain ouvre
  ces fichiers à chaque run (workflow nominal).
- **Correctif** : `config.excel_safe()` (préfixe `'` sur toute chaîne à premier caractère
  dangereux), appliqué à tous les writers CSV côté tri et drafts.
- **Test de fermeture** : ligne avec `nom="=2+2"` → la cellule exportée commence par `'=`.

### S2 — Import DB sans validation du format email — gravité MOYENNE — ✅ corrigé (51c79c3)
- **Constat** : `db.import_segments` accepte n'importe quelle chaîne comme email
  (aucun re-contrôle `EMAIL_REGEX`). Un CSV de segments forgé/altéré injecte des
  adresses malformées qui se propagent jusqu'à l'export ESP et au `.eml`.
- **Correctif** : re-validation `EMAIL_REGEX` après strip/lower ; lignes invalides
  ignorées et comptées (`skipped_invalid`), journalisées sans PII.
- **Test de fermeture** : import avec `pas-un-email` → 0 insertion, `skipped_invalid=1`.

### S3 — Header injection CRLF dans l'export .eml — gravité MOYENNE — ✅ corrigé (c22de3c)
- **Constat** : `sender.export_transport` construit `To:`/`Subject:` par f-string sans
  neutralisation. Défense en profondeur : une valeur DB contenant CR/LF injecterait
  des en-têtes (Bcc…) dans le fichier remis à l'ESP.
- **Correctif** : email contenant CR/LF → envoi refusé (compté `failed`) ; CR/LF du
  sujet repliés en espace.
- **Test de fermeture** : email `a@b.c\nBcc: x` → refus ; sujet multi-ligne → 1 seule ligne d'en-tête.

### S4 — Envoi réel possible avec placeholders non résolus — gravité HAUTE (RGPD) — ✅ corrigé (def9dca)
- **Constat** : si `.env` est incomplet, les corps contiennent `[LIEN_DESINSCRIPTION]`,
  `[VOTRE_LIEN_CALENDLY]`… `send_due(confirm=True)` les enverrait : **lien opt-out non
  fonctionnel** = violation RGPD directe + image désastreuse.
- **Correctif** : en envoi réel, tout message dont sujet/corps contient un placeholder
  par défaut de `MessageContext` est refusé (`skipped_placeholder`), sans event ni statut.
  Dry-run inchangé.
- **Test de fermeture** : envoi réel d'un message à placeholder → `sent=0`, `skipped_placeholder=1`.

### S5 — Redaction PII non récursive dans les logs JSON — gravité BASSE — ✅ corrigé (5eedcef)
- **Constat** : `_JsonFormatter` ne masque les clés PII qu'au 1er niveau du `context` ;
  un dict imbriqué `{"batch": {"email": …}}` fuiterait.
- **Correctif** : redaction récursive (dicts/listes).
- **Test de fermeture** : context imbriqué avec `email` → valeur masquée `***`.

## 🟠 UX / Optimisation

### U1 — `send --confirm` sans transport = dry-run silencieux — impact FORT / effort faible — ✅ (779c89b)
- **Constat** : `--confirm` sans `--export-dir` retombe en simulation ; l'opérateur
  croit avoir envoyé. Correctif : erreur CLI explicite (exit 2) exigeant un transport.
- **Test** : `send --confirm` sans transport → `SystemExit` + message explicite.

### U2 — `--day-index` warm-up manuel (défaut = plateau) — impact FORT / effort moyen — ✅ (96c2191)
- **Constat** : défaut `day_index=2` → plafond 100/j dès le 1er jour réel si l'opérateur
  oublie l'option : warm-up violé, délivrabilité du domaine en danger.
- **Correctif** : `infer_day_index()` = nb de jours distincts ayant déjà des events `sent`
  avant la date ; devient le défaut (0 historique → J1 → plafond 30). Option conservée.
- **Test** : sans historique → cap 30 ; après 2 jours d'envois → plateau.

### U3 — Dashboard : lisibilité + données manquantes — impact FORT / effort moyen — ✅ (9c0167c — agent graphiste)
- **Constat** : file de validation sans email ni date prévue ; aucune vue suppressions
  ni réponses par classe ; style minimal peu scannable pour l'usage quotidien.
- **Correctif** : colonnes email + `scheduled_at` ; sections « Suppressions par raison »
  et « Réponses par classe » ; refonte CSS (hiérarchie, KPI tiles, contraste) — HTML
  statique sans JS, délégué à un agent graphiste. Test : `tests/test_dashboard.py` (nouveau).

### U4 — `db stats` aveugle sur l'opérationnel — impact MOYEN / effort faible — ✅ (f8fd467)

## ⏭ Reportés (avec raison)
- **Heuristique `classify_reply` fragile** (`"oui"`/`"non"` en substring) : l'humain valide
  chaque classe avant action → risque contenu ; affiner demandera un corpus réel de réponses.
- **Re-plan de séquence ignorant les envois passés** (plafond théoriquement dépassable un
  jour déjà entamé) : le plafond est re-vérifié à l'envoi par `send_due` → non exploitable.
- **Zip-bomb XLSX** (openpyxl) : usage local par l'opérateur sur sa propre base ; hors modèle de menace V1.
